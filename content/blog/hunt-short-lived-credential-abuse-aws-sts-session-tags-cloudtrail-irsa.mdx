## Introduction
Short-lived AWS credentials are now the default path for both secure workloads and credential abuse, which makes “legit” API calls indistinguishable unless you reconstruct identity context end-to-end. The fastest way to reduce blind spots is to build a session lineage dataset that links **AWS STS** role assumption events to every downstream **CloudTrail** API call, enriched with session tags and OIDC/Kubernetes identity. Once lineage exists, you can enforce an allowlist for **EKS IRSA** subjects and alert on unexpected role chains into privileged roles. Skynet operationalizes this as standardized execution: deploy the normalization, enrichment, and detections across accounts/clusters and return prioritized findings with exact session identifiers in 24–48 hours.

## Quick Take
- Short-lived credentials aren’t inherently safe; without lineage, they erase the “who/what/where” you need for attribution.
- Treat **STS AssumeRole** and **AssumeRoleWithWebIdentity** as “auth events” and join them to subsequent **CloudTrail** activity by session keys.
- For **EKS IRSA**, your strongest signal is the OIDC `sub` claim (namespace/serviceAccount); allowlist it and alert on drift.
- High-risk detections: privileged role chains, sessions missing expected tags, and sudden changes in session geography/IP behavior.
- Ship the hunt as a repeatable pack: normalized schema, enrichment jobs, and a small set of high-signal queries.

## Build an STS Session Lineage Dataset (the prerequisite)
The core problem isn’t that adversaries use **STS**—it’s that many environments store **CloudTrail** as raw JSON blobs, query it inconsistently, and never connect the initial role assumption to the downstream API calls. Your first objective is a queryable schema where “session” becomes a first-class entity.

### Normalize CloudTrail into a queryable schema
At minimum, normalize fields you will join on repeatedly:
- `eventTime`, `eventSource`, `eventName`
- `recipientAccountId`, `awsRegion`, `sourceIPAddress`, `userAgent`
- `userIdentity.type`, `userIdentity.arn`, `userIdentity.principalId`
- `userIdentity.sessionContext.sessionIssuer.arn` (role ARN)
- `requestParameters`, `responseElements`

A practical pattern is **Amazon S3** for storage + **AWS Glue** + **Amazon Athena** for schema-on-read. If you already use **AWS Lake Formation**, align to it, but don’t block on it.

> ⚠️ Attention: If you only ingest management events but not data events for high-impact services (for example, **Amazon S3** object access, **AWS KMS** decrypt usage), lineage may “look clean” while impact happens elsewhere.

### Extract the “assumption events” as your root nodes
Start by pulling all `AssumeRole*` events; these are your session roots.

```sql
-- Athena example: root STS events
SELECT
  eventTime,
  eventName,
  recipientAccountId,
  awsRegion,
  sourceIPAddress,
  userAgent,
  userIdentity.type                 AS caller_type,
  userIdentity.arn                  AS caller_arn,
  userIdentity.principalId          AS caller_principalid,
  userIdentity.sessionContext.sessionIssuer.arn AS assumed_role_arn,
  requestParameters                 AS req,
  responseElements                  AS resp
FROM cloudtrail
WHERE eventSource = 'sts.amazonaws.com'
  AND eventName IN ('AssumeRole','AssumeRoleWithWebIdentity','AssumeRoleWithSAML')
  AND eventTime >= date_add('day', -7, current_timestamp);
```

### Join downstream API calls to the STS session
For downstream events, the join keys you can reliably use depend on the caller type:
- For assumed-role activity, `userIdentity.principalId` typically contains a stable role-session identifier (often `ROLEID:RoleSessionName`).
- `userIdentity.arn` becomes an `assumed-role/.../RoleSessionName` ARN.
- **CloudTrail** also emits session context (issuer role ARN) that can be used for grouping.

A pragmatic “lineage view” is: (1) capture root assume-role events; (2) capture all events where `userIdentity.type = 'AssumedRole'`; (3) join by `recipientAccountId + principalId` and time window.

```sql
-- Athena example: build a session-centric view
WITH sts_roots AS (
  SELECT
    recipientAccountId AS account_id,
    eventTime          AS sts_time,
    eventName          AS sts_event,
    sourceIPAddress    AS sts_ip,
    userAgent          AS sts_ua,
    userIdentity.principalId AS caller_principalid,
    userIdentity.arn   AS caller_arn,
    userIdentity.sessionContext.sessionIssuer.arn AS role_arn,
    json_extract_scalar(responseElements, '$.credentials.accessKeyId') AS issued_access_key
  FROM cloudtrail
  WHERE eventSource='sts.amazonaws.com'
    AND eventName IN ('AssumeRole','AssumeRoleWithWebIdentity')
),
api_calls AS (
  SELECT
    recipientAccountId AS account_id,
    eventTime,
    eventSource,
    eventName,
    awsRegion,
    sourceIPAddress,
    userAgent,
    userIdentity.type AS identity_type,
    userIdentity.principalId AS principalid,
    userIdentity.sessionContext.sessionIssuer.arn AS role_arn
  FROM cloudtrail
  WHERE userIdentity.type IN ('AssumedRole')
)
SELECT
  a.account_id,
  r.role_arn,
  a.principalid,
  min(r.sts_time) AS first_seen_assume,
  max(a.eventTime) AS last_seen_api,
  approx_distinct(a.sourceIPAddress) AS distinct_ips,
  approx_distinct(a.awsRegion) AS distinct_regions,
  count(*) AS api_call_count
FROM api_calls a
LEFT JOIN sts_roots r
  ON a.account_id = r.account_id
 AND a.role_arn = r.role_arn
 AND a.eventTime >= r.sts_time
 AND a.eventTime <= r.sts_time + interval '12' hour
GROUP BY 1,2,3;
```

> 💡 Pro-Tip: Persist lineage outputs as a daily partitioned table (or view backed by CTAS). When incidents happen, you want “session to blast radius” queries to run in seconds, not minutes.

## Enrich Lineage with STS Session Tags and OIDC Claims
Lineage without identity enrichment still forces manual interpretation. Your next objective is to answer: “Which workload identity (or human) produced this role session?” For IRSA-backed sessions, that means mapping OIDC subject to a Kubernetes service account and cluster.

### Capture session tags (and alert on absence)
Session tags are one of the cleanest ways to carry execution context into **STS** sessions, but many teams don’t enforce them. If your platform relies on tags like `env`, `app`, `repo`, `cluster`, or `owner`, missing tags become a detection.

Key places tags appear:
- `requestParameters.tags` (for `AssumeRole` with session tags)
- Depending on event type and client, tags may appear nested; validate against your actual **CloudTrail** records.

```sql
-- Athena example: identify sessions missing expected tags
SELECT
  eventTime,
  userIdentity.sessionContext.sessionIssuer.arn AS role_arn,
  userIdentity.principalId AS principalid,
  sourceIPAddress,
  userAgent,
  requestParameters
FROM cloudtrail
WHERE eventSource='sts.amazonaws.com'
  AND eventName='AssumeRole'
  AND (
    json_extract(requestParameters, '$.tags') IS NULL
    OR json_array_length(cast(json_extract(requestParameters, '$.tags') as array(json))) = 0
  );
```

> ⚠️ Attention: Don’t treat “missing tags” as malicious by default. Treat it as “unattributed execution context,” then prioritize based on role sensitivity and downstream actions.

### Extract OIDC subject from AssumeRoleWithWebIdentity
For **EKS IRSA**, the decisive attribute is the OIDC `sub` claim, typically shaped like:
- `system:serviceaccount:<namespace>:<serviceaccount>`

In **CloudTrail**, `AssumeRoleWithWebIdentity` includes a `webIdentityToken` reference (not the raw token) and related context. Your enrichment job should parse out the subject/issuer when available in the event record, and otherwise correlate by role ARN + known trust policy + Kubernetes inventory (next section).

> 💡 Pro-Tip: If you control the calling code, set `RoleSessionName` deterministically (for example, include workload and pod UID). That turns role sessions into high-fidelity pivot points.

## Validate IRSA: Detect Impossible or Drifted Workload Identity
IRSA abuse often looks like “normal” role usage because it is normal role usage—just from the wrong service account, namespace, or cluster. The fastest high-signal control is to maintain an allowlist of expected mappings:
- Kubernetes `namespace/serviceAccount` → IAM role ARN
- Cluster identifier (or OIDC provider) → allowed subjects

### Inventory service accounts and bound role ARNs
Pull the authoritative mapping directly from the cluster.

```bash
kubectl get sa -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}{"\t"}{.metadata.annotations.eks\.amazonaws\.com/role-arn}{"\n"}{end}'
```

Store this output (or API equivalent) in a table keyed by cluster + namespace + serviceAccount.

### Alert on new or unexpected OIDC subjects
Once you can extract (or infer) `sub` from `AssumeRoleWithWebIdentity`, compare it to the allowlist for that role and cluster/OIDC provider.

Detection logic patterns that work well:
- New `sub` values for a role (first-seen)
- Known `sub` used from an unexpected cluster/OIDC provider
- A role assumed via web identity where the role is expected to be assumed only by `AssumeRole` (or vice versa)

> ✅ Success: When this is wired correctly, you can go from “this role was used” to “this exact namespace/serviceAccount in this cluster produced the calls,” which sharply reduces investigation time.

## Flag High-Impact STS Chains and “Legit” Abuse Patterns
With lineage + enrichment, prioritize what matters: role sessions that increase privilege, expand blast radius, or break expected invariants.

### Detect privileged role chains (role-to-role escalation)
The most common high-impact chain is:
- initial foothold (workload identity) → intermediate role → privileged role

You can approximate this by looking for sequences where:
- A role session performs **STS** calls to assume another role
- The target role is in a high-sensitivity list (admin, security tooling, key management, org management)

```sql
-- Athena example: find role sessions that call STS AssumeRole into sensitive roles
WITH role_calls_sts AS (
  SELECT
    eventTime,
    recipientAccountId AS account_id,
    userIdentity.principalId AS src_principalid,
    userIdentity.sessionContext.sessionIssuer.arn AS src_role_arn,
    json_extract_scalar(requestParameters, '$.roleArn') AS dst_role_arn,
    sourceIPAddress,
    userAgent
  FROM cloudtrail
  WHERE eventSource='sts.amazonaws.com'
    AND eventName='AssumeRole'
    AND userIdentity.type='AssumedRole'
)
SELECT *
FROM role_calls_sts
WHERE dst_role_arn LIKE '%Admin%'
   OR dst_role_arn LIKE '%Organization%'
   OR dst_role_arn LIKE '%Security%';
```

> ⚠️ Attention: Name-based matching is a stopgap. Replace it with an explicit sensitivity catalog: role ARN → tier (0–3) based on attached policies, permission boundaries, and reachable resources.

### Spot anomalous session behavior
Once you have session-level aggregates, anomaly checks become straightforward and fast:
- Same `principalId` seen across many source IPs in a short window
- Sudden region spread (`distinct_regions`) for a session
- New user agent for a known workload role
- API mix changes (for example, a workload role suddenly calls **AWS KMS**, **IAM**, **Organizations**, **EC2** network changes)

Keep it pragmatic: you don’t need complex models; you need deterministic tripwires plus “first-seen” conditions.

### Prioritize findings by blast radius
Rank sessions using a few concrete inputs:
- Role sensitivity tier
- Downstream calls to high-impact services (**AWS IAM**, **AWS KMS**, **AWS Organizations**, **Amazon S3**, **Amazon ECR**, **AWS Secrets Manager**)
- Cross-account access (role in Account A used to act in Account B)
- Persistence indicators (creation of new roles, access keys, policy updates)

## Checklist
- [ ] Confirm **CloudTrail** is enabled org-wide and delivered to a central **Amazon S3** bucket with immutable retention controls.
- [ ] Ensure management events are complete; add data events for high-impact services you actually care about (for example, **Amazon S3**, **AWS KMS**).

- [ ] Build a session lineage view joining `AssumeRole*` roots to `AssumedRole` downstream API calls.
- [ ] Define required session tags for workload roles and alert on missing tags for sensitive roles.
- [ ] Inventory **EKS** service accounts annotated with `eks.amazonaws.com/role-arn` across all clusters.
- [ ] Maintain an allowlist: OIDC provider + `sub` → approved namespace/serviceAccount per role.
- [ ] Detect role-to-role chains into sensitive roles and prioritize by sensitivity tier.
- [ ] Create an investigation playbook: pivot from `principalId` → full API timeline → impacted resources → remediation.

## FAQ
### What’s the single highest-signal pivot in CloudTrail for short-lived credential abuse?
<p>Session lineage. Treat STS role assumptions as authentication events, then pivot on the assumed-role session (principalId/role session) to enumerate every downstream API call with consistent enrichment (role issuer, tags, and web identity context).</p>

### How do we reduce false positives when alerting on IRSA subject changes?
<p>Anchor alerts to an explicit allowlist per cluster/OIDC provider and role ARN. Then scope “new subject” alerts to sensitive roles or to subjects that immediately perform high-impact actions (IAM/KMS/Organizations/network changes).</p>

### What should the output of a standardized hunt look like for leadership and engineers?
<p>Two layers: (1) an executive summary of affected roles, accounts, and blast radius; (2) an engineer-ready findings set keyed by exact session identifiers (principalId/role ARN), timestamps, and the minimal remediation steps (tighten trust policies, rotate impacted secrets, fix IRSA bindings, enforce session tags).</p>
